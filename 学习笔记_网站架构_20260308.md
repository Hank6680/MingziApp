# 网站架构学习笔记
> 日期：2026-03-08 | 项目：MingziApp

---

## 一、整体架构

```
浏览器 (React)              服务器 (Node.js)           数据库 (SQLite)
──────────────             ──────────────────         ──────────────
src/pages/                 server/index.js             data.db
src/components/            server/routes/
src/context/               server/middleware/
src/api/client.ts          server/db.js
```

**两个独立进程，唯一联系是 HTTP 请求。**  
前端是静态文件（HTML+JS），后端是独立 Node.js 进程，部署在不同地方。

## 二、前端结构

| 目录/文件 | 职责 |
|---|---|
| `pages/` | 每个页面一个文件，决定页面长什么样、怎么交互 |
| `components/` | 可复用 UI 块（Navbar、Badge），多页面共用 |
| `context/` | 全局状态（登录用户、购物车、Toast），任何页面都能读 |
| `api/client.ts` | 所有 HTTP 请求的封装，统一加 token header |
| `App.tsx` | 路由配置，决定哪个 URL 渲染哪个页面 |

### Context（全局公告板）
- 问题：数据默认只能从父组件传子组件（props），但登录用户信息所有页面都要用
- 解决：`AuthProvider` 包在最外层，里面所有组件通过 `useAuth()` 直接读取
- 类比：餐厅门口的菜单公告板，所有服务员进门就能看到

### ProtectedRoute（门卫组件）
```tsx
// 逻辑：没登录 → 去登录页；没权限 → 去默认页；通过 → 放行
if (!user) return <Navigate to="/login" />
if (roles && !roles.includes(user.role)) return <Navigate to="/products" />
return children
```

### 组件挂载
- 挂载 = 组件**第一次出现在屏幕上**的那一刻
- 卸载 = 组件从屏幕消失（切换页面）
- `useEffect(fn, [])` = 挂载时执行一次，`[]` 表示没有依赖，不会重复执行
- 没有 `[]` 会无限循环：执行 → setXxx → 重新渲染 → 再执行…

### localStorage（登录态持久化）
```js
// 登录成功 → 存入
localStorage.setItem('mingzi-auth', JSON.stringify(data))

// App 启动 → 读取恢复状态（useEffect [] 只跑一次）
const raw = localStorage.getItem('mingzi-auth')

// 退出 → 清除
localStorage.removeItem('mingzi-auth')
```
刷新页面不会退出，因为 token 存在浏览器本地。

## 三、一次完整请求的链路

以"创建订单"为例：

```
1. 前端 StaffOrderingPage 点击提交
   └── api/client.ts: POST http://localhost:4000/api/orders + body + token

2. server/index.js 收到请求
   └── CORS检查（来源是否合法）
   └── express.json() 解析 body → req.body
   └── 转发给 routes/orders.js

3. middleware/auth.js 验证 token
   └── 从 header 取出 Bearer token
   └── jwt.verify() 解密
   └── req.user = { id, role, username }
   └── next() 放行，或 next(401) 拦截

4. routes/orders.js 处理业务逻辑
   └── 取 req.body 数据
   └── 验证格式
   └── dbRun("INSERT INTO orders ...") 写入数据库
   └── res.status(201).json(newOrder) 返回结果

5. 前端收到响应
   └── 更新 state → React 重新渲染界面
```

## 四、后端结构

| 文件 | 职责 |
|---|---|
| `index.js` | 启动服务器，注册所有路由，CORS 和全局错误处理 |
| `db.js` | 初始化 SQLite 连接，建表（服务器首次启动时执行） |
| `routes/orders.js` | 订单的增删改查逻辑 |
| `middleware/auth.js` | 验证 JWT token，检查角色权限 |
| `utils/httpError.js` | 统一创建错误对象 |

### 中间件链（请求经过的流水线）
```
请求 → cors → json解析 → auth验证 → 具体route → 错误处理 → 响应
```
任意一环调用 `next(error)` 就跳到最后的错误处理，不再往下走。

## 五、数据库（SQLite）

### 三个操作函数
| 函数 | 用途 | 返回 |
|---|---|---|
| `dbRun` | INSERT / UPDATE / DELETE（写入） | 操作结果（lastID 等） |
| `dbAll` | SELECT 多行 | 数组 `[{...}, {...}]` |
| `dbGet` | SELECT 单行 | 一个对象 `{...}` 或 null |

### SQL 四个核心操作
```sql
SELECT * FROM orders WHERE customerId = 5      -- 查
INSERT INTO orders (customerId, status) VALUES (5, 'created')  -- 写入
UPDATE orders SET status = 'shipped' WHERE id = 1              -- 修改
DELETE FROM orders WHERE id = 3                                 -- 删除
```

### ? 占位符（防 SQL 注入）
```js
// 危险：直接拼字符串，用户可以输入 ' OR '1'='1 绕过验证
db.run(`SELECT * FROM users WHERE username = '${username}'`)

// 安全：? 让 SQLite 把输入当纯数据处理，不会被解析成 SQL 指令
db.run("SELECT * FROM users WHERE username = ?", [username])
```

### JOIN（关联查询）
数据分表存，`orders` 只存 `customerId`，客户名在 `customers` 表。

```sql
SELECT o.*, c.name as customerName
FROM orders o
LEFT JOIN customers c ON c.id = o.customerId
```
- `ON` 后面写拼接条件
- `LEFT JOIN`：左表数据一定显示，右表没匹配填 NULL（数据不丢）
- `INNER JOIN`：两边都有才显示（数据可能丢失）

## 六、HTTP 状态码

| 开头 | 含义 | 常见码 |
|---|---|---|
| 2xx | 成功 | 200 OK / 201 Created / 204 No Content |
| 4xx | 客户端的错 | 400 参数错误 / 401 未登录 / 403 无权限 / 404 找不到 |
| 5xx | 服务器的错 | 500 服务器崩了 |

- **401 vs 403**：401 = 不知道你是谁（没 token）；403 = 知道你是谁，但不让进
- `res.ok` = 状态码在 200-299 之间为 true，其他都 throw Error

## 七、浏览器 DevTools（F12）

| 面板 | 用途 |
|---|---|
| **Console** | 看 console.log，看 JS 报错（红色）。前端出问题先看这里 |
| **Network** | 看所有 HTTP 请求和响应。调 API 出问题先看这里 |
| **Application** | 看 localStorage（存着 token 和用户信息） |
| **Elements** | 看页面 HTML 结构和 CSS |

### Network 面板各列
- **Name**：请求路径（`orders` = GET /api/orders）
- **Status**：状态码，红色=出错
- **Type**：`fetch` = 代码发的请求；`preflight` = 浏览器自动发的 CORS 预检，不用管
- **Time**：请求耗时

### 调试技巧
点开一条 fetch 请求 → **Response 标签** = 服务器返回的原始 JSON，出 bug 必看。

## 八、实时同步（扩展知识）

需求：手机A下单，电脑B拣货页自动刷新

**方案1：轮询（当前可实现）**
```js
useEffect(() => {
  fetchOrders()
  const timer = setInterval(() => fetchOrders(), 10000)  // 每10秒刷一次
  return () => clearInterval(timer)  // 页面关闭时停掉定时器
}, [])
```

**方案2：WebSocket（后端主动推送，更高效，但复杂）**  
有新数据时后端主动通知前端，不需要定时轮询。

## 十、嵌入式 → Web 类比速查（给有嵌入式背景的人）

### 架构对应关系

| 嵌入式 | Web 对应 | 说明 |
|--------|----------|------|
| 单片机主控（MCU） | 服务器（Node.js） | 核心业务逻辑在这里跑 |
| 传感器 / 输入设备 | 前端页面（React） | 采集用户输入，发给主控 |
| 执行器 / 输出设备 | 浏览器渲染结果 | 把处理结果呈现给用户 |
| 串口 / SPI / I2C | HTTP API | 模块间传数据的协议 |
| EEPROM / Flash | 数据库（SQLite） | 断电不丢失的持久化存储 |
| RAM | 缓存（内存变量） | 快速读写，进程重启后丢失 |
| 中断（Interrupt） | 事件 / Webhook | 外部触发，不需要轮询 |
| 轮询（Polling） | 前端定时请求 | 主动定期去问"有没有新数据" |
| 固件烧录 | 部署（Deploy） | 把新版本推上去替换旧版本 |
| 看门狗（Watchdog） | 健康检查 / 进程守护 | 程序崩了自动重启 |
| 中断向量表 | routes/ 路由表 | 不同"信号"路由到不同处理函数 |
| 信号预处理电路 | middleware/ 中间件 | 请求进来先过滤 / 验证 |
| 串口驱动库 | api/client.ts | 封装好的通信接口，直接调用 |

### MingziApp 具体对应

```
src/（React前端）      = LCD 显示模块，只管呈现
server/（Node.js）    = MCU 主控，处理业务逻辑
server/db.js          = EEPROM，持久化存数据
server/routes/        = 中断向量表，不同请求路由到不同处理函数
server/middleware/    = 信号预处理电路，请求进来先过滤/验证
api/client.ts         = 封装好的"串口驱动库"，前端用它和后端通信
```

### 通信协议类比

```
串口通信：                          HTTP 请求：
  双方约定波特率（速率）               双方约定 HTTP 协议
  发送方打包数据帧                     客户端发请求报文
  接收方按格式解析                     服务器按格式解析
  有起始位、数据位、校验位             有请求行、Header、Body
```
本质一样——**双方提前约定格式，按格式打包/解包数据。**

### 高并发 = 软件版中断机制

```
嵌入式非阻塞：
  发出 SPI 读取命令，不死等
  主循环继续跑别的任务
  SPI 传输完成 → 中断触发 → 回来处理数据

Node.js 事件循环：
  收到请求A → 发出数据库查询，不等结果，继续
  收到请求B → 发出数据库查询，不等结果，继续
  数据库返回A的结果 → 事件触发 → 处理，发回响应
```

`async/await` 的含义：「这里要等 IO，先去处理别的，等好了再回来。」

### 嵌入式 vs Web 的本质差异

| | 嵌入式 | Web 服务器 |
|--|--------|-----------|
| 任务模型 | 单任务 / 简单多任务 | 高并发，同时处理成千上万请求 |
| 资源约束 | RAM 极度有限（KB级） | 内存相对充裕（GB级） |
| 实时性要求 | 微秒级确定性 | 毫秒级响应，允许抖动 |
| 语言选择 | C/C++（无运行时） | JS/Python/Go（有运行时） |
